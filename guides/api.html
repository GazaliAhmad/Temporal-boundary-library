Spec

Current API

The current 3.x API makes boundary-defined windows explicit: a boundary event opens or closes a meaningful span of time, and the library resolves those windows with Temporal and explicit IANA time zones.

Design Goals

North star
  • Use Temporal as the core time model.
  • Require explicit IANA time zones.
  • Represent windows as exact time in a named zone.
  • Handle DST and non-24-hour days correctly.
  • Keep calendar labeling out of the core library.
  • Preserve the main mental model: boundary events define windows.
  • Keep the API idiomatic to Temporal instead of recreating parallel date/time concepts.

Non-goals

Intentional boundaries
  • Calculating religious calendars.
  • Calculating lunar calendar labels.
  • Embedding domain-specific boundary meaning into the API.
  • Hiding time zone handling behind host-local defaults.

Core Concepts

Mental model

Boundary window

A boundary event is a resolved point in time with domain meaning, such as a cutoff, changeover, opening, closing, start, or end. A boundary window is the interval between two boundary events: 

[boundary_n, boundary_n+1)

Boundaries are resolved in an explicit IANA time zone and returned as Temporal.ZonedDateTime.

Time model

  • Exact time: Temporal.Instant
  • Wall-clock local date/time: Temporal.PlainDateTime
  • Zone-aware exact/local value: Temporal.ZonedDateTime

Core comparison, grouping, and progress calculations use exact time semantics. 

type BoundaryWindow = {
  start: Temporal.ZonedDateTime;
  end: Temporal.ZonedDateTime;
  label: string;
  metadata: Record<string, unknown>;
};

Package Entry and Strategy API

Public shape

Package entry

import { ... } from 'day-boundary';

BoundaryStrategy

interface BoundaryStrategy {
  timeZone: string;
  getWindowForInstant(instant: Temporal.Instant): BoundaryWindow;
}
  • Every strategy must have an explicit timeZone.
  • The primary resolution entry point is getWindowForInstant.
  • Strategies may internally resolve local dates, but public resolution stays exact-time based.

FixedTimeBoundaryStrategy

Built-in strategy

Use this when the boundary is the same wall-clock time every day in a given zone. 

new FixedTimeBoundaryStrategy({
  timeZone: string,
  boundaryTime: Temporal.PlainTime,
  label?: string,
  disambiguation?: 'compatible' | 'earlier' | 'later' | 'reject',
})
const strategy = new FixedTimeBoundaryStrategy({
  timeZone: 'Europe/London',
  boundaryTime: Temporal.PlainTime.from('09:00'),
  label: 'operational-day',
});
  • Boundary time is interpreted as local wall-clock time in the configured zone.
  • DST ambiguity is resolved using disambiguation.
  • The duration between boundaries is not assumed to be 24 hours.

DailyBoundaryStrategy

Built-in strategy

Use this when the boundary changes by date and must come from a timetable or resolver. 

new DailyBoundaryStrategy({
  timeZone: string,
  label?: string,
  disambiguation?: 'compatible' | 'earlier' | 'later' | 'reject',
  getBoundaryForDate: (
    date: Temporal.PlainDate,
    context: {
      timeZone: string,
      disambiguation: 'compatible' | 'earlier' | 'later' | 'reject',
      calendar: 'iso8601',
    }
  ) => Temporal.ZonedDateTime,
})
const strategy = new DailyBoundaryStrategy({
  timeZone: 'Asia/Singapore',
  label: 'shifting-day',
  getBoundaryForDate(date, context) {
    const key = date.toString();
    const boundaryTime = boundaryByDate[key];
    const time = Temporal.PlainTime.from(boundaryTime);

    return Temporal.ZonedDateTime.from({
      timeZone: context.timeZone,
      year: date.year,
      month: date.month,
      day: date.day,
      hour: time.hour,
      minute: time.minute,
    });
  },
});
  • The resolver must support the dates needed to derive the surrounding window.
  • The strategy resolves previous, current, and next local dates around the target instant.
  • The returned boundary should stay in context.timeZone.

DST and Time Zone Behavior

Explicit by design
  • Supports IANA time zones such as Asia/Singapore, Europe/London, and America/New_York.
  • Handles DST transitions, repeated local times, and skipped local times.
  • Never depends on the host machine's implicit local time zone.
  • Never assumes a day is 24 hours.
  • Public APIs avoid manual offset math and rely on Temporal's zone resolution behavior.

Error Behavior

Failure modes
  • Throws when a strategy is invalid.
  • Throws when a required time zone is missing or invalid.
  • Throws when a boundary cannot be resolved.
  • Throws when resolved boundaries do not form a valid window.
  • Throws when a local date/time cannot be resolved and disambiguation: 'reject' is used.

Boundary Duration Semantics

Elapsed vs wall-clock

For worker schedules, schools, delivery operations, and factory processes, the API keeps two meanings separate: elapsed duration and wall-clock schedule. These diverge on DST transition days.

  • Temporal.Instant is the right primitive for elapsed-duration rules.
  • Temporal.ZonedDateTime is the right primitive for wall-clock schedule rules.
  • This distinction should stay explicit in examples, docs, and higher-level helpers.
const start = Temporal.ZonedDateTime.from(
  '2026-10-25T00:00:00+01:00[Europe/London]'
);

const result = compareWindowEndings(start, { hours: 8 });

// elapsedEnd   -> 2026-10-25T07:00:00+00:00[Europe/London]
// wallClockEnd -> 2026-10-25T08:00:00+00:00[Europe/London]

Boundary duration helpers

  • getWindowEndByElapsedDuration(start, duration)
  • getWindowEndByWallClockDuration(start, duration)
  • compareWindowEndings(start, duration)

Calendar Labeling Policy

Out of scope

Calendar labeling is intentionally outside the core library. The library resolves windows; applications or companion modules map those windows to labels.

  • Hijri calendar datasets
  • Chinese lunar calendar datasets
  • Business reporting day labels
  • Custom user-defined day names

Migration Guidance from v1

Upgrade path

Conceptual changes

  • v1 used Date and implicit host-local behavior.
  • The current API uses Temporal and explicit IANA time zones.
  • Windows are resolved using exact-time semantics.
  • The core surface prefers Temporal-native types.

Mapping

  • getActiveWindow(now, strategy) -> getWindowForInstant(Temporal.Now.instant(), strategy)
  • getWindowForTimestamp(timestamp, strategy) -> getWindowForInstant(...) or getWindowForPlainDateTime(...)
  • FixedTimeBoundaryStrategy still exists conceptually but now requires timeZone.
  • DailyBoundaryStrategy now resolves boundaries with Temporal.

Common Legacy Inputs and Explicit Failures

Runtime guidance

FixedTimeBoundaryStrategy

Legacy fixed-time keys such as startHour, startMinute, hour, minute, and second now throw targeted migration errors.

new FixedTimeBoundaryStrategy({
  timeZone: 'Asia/Singapore',
  boundaryTime: '06:00',
});

DailyBoundaryStrategy

Missing timeZone when a resolver is present now throws an explicit migration error, and legacy resolver return values such as Date, string, and number are rejected.

new DailyBoundaryStrategy({
  timeZone: 'Asia/Singapore',
  getBoundaryForDate(date, context) {
    return Temporal.ZonedDateTime.from({
      timeZone: context.timeZone,
      year: date.year,
      month: date.month,
      day: date.day,
      hour: 19,
      minute: 0,
    });
  },
});

Exact-time inputs

  • Legacy Date, string, and numeric timestamp inputs throw targeted migration errors.
  • null and undefined exact-time inputs throw typed runtime errors expecting Temporal.Instant or Temporal.ZonedDateTime.
const instant = Temporal.Instant.from('2026-04-19T01:00:00Z');
const window = getWindowForInstant(instant, strategy);

Wall-clock local inputs

  • Legacy Date, string, and numeric inputs throw targeted migration errors.
  • null and undefined wall-clock inputs throw typed runtime errors expecting Temporal.PlainDateTime.
  • boundaryTime: '' throws a library-owned validation error.
  • In TypeScript, boundaryTime is typed as string | Temporal.PlainTime, so empty strings remain representable and are rejected at runtime.
const localInput = Temporal.PlainDateTime.from('2026-10-25T01:30:00');
const window = getWindowForPlainDateTime(localInput, strategy);

Removed or renamed entry points

  • day-boundary/shifts -> root day-boundary entry
  • compareShiftEndings(...) -> compareWindowEndings(...)
  • getShiftEndByElapsedDuration(...) -> getWindowEndByElapsedDuration(...)
  • getShiftEndByWallClockDuration(...) -> getWindowEndByWallClockDuration(...)

Design Decisions and Current Recommendation

Current line
  1. BoundaryWindow.start and BoundaryWindow.end use Temporal.ZonedDateTime.
  2. Exact comparisons and duration math use toInstant() semantics internally.
  3. The public API prefers readability and explicit zone context over a lower-level Instant + timeZone window shape.
  • Core window values use Temporal.ZonedDateTime.
  • Exact comparisons use toInstant().
  • Input helpers accept Temporal objects only.
  • Zone and DST handling are explicit.
  • Calendar labeling remains outside core.
